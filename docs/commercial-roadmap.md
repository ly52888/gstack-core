# Commercial Roadmap

## Strategy

Use open source to prove the router, earn trust, and attract capability-card contributions. Monetize packaged expertise, private setup, governance, analytics, and maintained workflow packs.

The market wedge is Codex workflow maturity: as more builders use Codex for real projects, they need repeatable routing, verification, recovery, external-AI disclosure, and team governance rather than another prompt collection.

## Open Core

Always free:

- Core `gstack-skills` router.
- Clarification gate.
- Natural command routing.
- Multi-AI council workflow.
- Capability card spec.
- Doctor checks.
- Route trace format.
- Bundled codegraph-lite.
- Basic eval prompts.

Keeping these free makes the project credible. The commercial layer should help teams apply the router reliably across many projects, people, and policies.

## Paid Layers

### Pro Workflow Packs

Sell curated, maintained packs for high-value workflows:

- Developer Pack: implementation, debugging, review, test generation, closeout.
- GitHub Pack: PR triage, CI failure repair, issue-to-plan, release notes.
- Creative Pack: image/video/design routing, brand gates, asset verification.
- Website Build Pack: reference-site intake, multi-AI council, differentiated spec, implementation handoff.
- Local-Prod Pack: runtime recovery, health checks, logs, rollback, handoff.
- Security Pack: auth/payment/data-risk workflows, external AI disclosure, secret handling.

### Team Setup

Service offering for teams:

- Audit current Codex usage.
- Build project capability cards.
- Add team policy and allowlists.
- Define route trace and verification standards.
- Create project profiles and handoff templates.
- Train team members on natural command patterns.

### Enterprise Governance

Potential product layer:

- Private capability registry.
- Tool/plugin allowlists and denylists.
- External AI disclosure policy.
- Route trace dashboard.
- Verification and closeout reports.
- Pack versioning and rollout controls.
- Organization profiles.
- Audit export for regulated workflows.

## Early Pricing Hypotheses

These are planning anchors, not public pricing:

- Pro Packs: one-time or annual license per user/team.
- Team Setup: fixed implementation package.
- Enterprise: annual subscription based on seats, repositories, or managed profiles.
- Support: paid priority support and custom workflow design.

## Phased Plan

### Phase 0: Public Credibility

- Publish open core.
- Ship examples and eval prompts.
- Show natural command demos.
- Show reference-site multi-AI council demo.
- Collect issues and capability-card requests.

### Phase 1: Packaged Expertise

- Release first Pro Pack as a separate private or paid distribution.
- Offer team setup manually.
- Use real user workflows to improve open-core extension points.

### Phase 2: Governance

- Add private registry format.
- Add policy files and validation.
- Add route trace summaries that can feed dashboards.
- Add pack versioning.

### Phase 3: Managed Platform

- Cloud sync for private registries.
- Team analytics.
- Policy enforcement.
- Marketplace or pack catalog.

## Risks

- Overbuilding SaaS before the workflow is trusted.
- Weak open core that feels like a teaser.
- Vague "AI automation" positioning.
- Legal risk if website inspiration is framed as cloning.
- Enterprise trust risk if external AI disclosure is not explicit.

## Current Recommendation

Publish the open core first. Commercialize through services and packs before building SaaS.

## Near-Term Marketing Actions

- Ship a short demo for each natural command pattern.
- Publish before/after examples: vague prompt versus routed gstack workflow.
- Show the multi-AI council workflow on a public reference website.
- Show a failed page task moving from screenshot to root cause to verified fix.
- Show route trace as the trust artifact.
- Collect early users through GitHub issues and discussions before selling software.
